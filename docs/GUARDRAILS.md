# YAT YAS UP — Safety and Abstraction Guardrails

## Purpose

This document protects the project from drifting into real-world tactical instruction.

YAT YAS UP may use public planning themes and broad operational concepts as inspiration, but all mechanics must remain abstract, game-safe, and tabletop-oriented.

## Core rule

If a mechanic would teach a player how to perform a real-world task, it does not belong in the game.

## Allowed abstraction layer

Allowed mechanics may model:

- pressure,
- friction,
- delay,
- readiness,
- command continuity,
- abstract recovery queue,
- support marker effect,
- control-point progression,
- phase gates,
- AAR categories,
- plan-vs-actual differences.

## Not allowed

Do not model or describe:

- real tactics,
- real fire support procedures,
- real weapon employment,
- targeting methods,
- vulnerability exploitation,
- breach methods,
- real-world obstacle reduction,
- actual communications procedures,
- actual frequencies, nets, or radio practices,
- real maintenance procedures,
- diagnostic steps,
- vehicle component failure details,
- casualty simulation,
- squad battle drills,
- field-manual procedures,
- step-by-step operational methods.

## Fire Support and MG Base guardrail

Locked rule:

- MG Base reduces abstract pressure marker effect.
- Fire Support reduces abstract pressure marker effect.
- Neither action moves vehicles.
- Neither action models targeting.
- Neither action models weapon behavior.
- Neither action models real effects.

Use language like:

> Fire Support reduces pressure in the selected marker.

Do not use language like:

> The mortar targets / destroys / neutralizes [specific real-world target].

## Dismount / screening guardrail

Locked rule:

- Dismounted bundles reduce pressure-marker effects in nearby cells.
- Mounted movement through unscreened pressure is penalized.
- Mounted movement through screened pressure is less penalized.

Do not justify this with real tactical method descriptions.

Use language like:

> Screening reduces pressure effect around the dismount point.

Do not describe how dismounted units accomplish it.

## Recovery guardrail

Locked rule:

- PSG coordinates recovery.
- R7 and parts track perform recovery/repair route.
- Repairs are abstract readiness restoration over time.

Do not model:

- component-level repairs,
- fault diagnosis,
- recovery procedures,
- vehicle vulnerabilities,
- realistic tow methods,
- maintenance echelons beyond a broad abstract distinction if needed.

## Command/control guardrail

Locked rule:

- C2 Net is represented by a game-safe tether value.
- Order delay is abstract latency.

Do not model:

- real communications procedures,
- radio nets,
- frequencies,
- call signs,
- electronic warfare methods,
- actual command relationships beyond game labels.

## Language guidance

Preferred terms:

- pressure marker
- friction marker
- screening
- readiness
- degraded
- recovery route
- repair cycle
- C2 tether
- command continuity
- abstract support
- AAR recommendation

Avoid terms that imply real combat procedure or direct tactical instruction.

## Design review test

Before adding any feature, ask:

1. Is this a board-game abstraction?
2. Does it avoid teaching a real procedure?
3. Does it improve planning/control gameplay?
4. Does it preserve the YAT YAS identity?
5. Does it keep Fire Support and MG Base pressure-only?
6. Does it avoid unnecessary realism?

If any answer is no, redesign the mechanic.
