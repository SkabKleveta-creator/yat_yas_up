# YAT YAS UP — Project Closeout

**Date:** 2026-06-20  
**Project lane:** Planning/preparation TTRPG prototype  
**Current prototype:** v0.3.5  
**Repository:** `SkabKleveta-creator/yat_yas_up`

## BLUF

Today moved **YAT YAS UP** from a loose AAV-themed prototype into a governed planning/control board with a clear design identity, documented guardrails, a locked v0.3 logic spine, and a repository documentation set. The core product is now best described as a **planning/preparation TTRPG trial engine** where the player selects tracks or markers, issues phase-valid orders, manages abstract pressure and recovery, then uses the AAR to improve the next run.

## What changed today

### 1. Design direction clarified

The project is no longer being treated as a conventional arcade mini-game.

Locked direction:

```text
PLAN → EXECUTE → DISRUPT → RECOVER → LEARN
```

YAT YAS UP is now framed as:

- a planning and control board,
- a tabletop-style mission rehearsal prototype,
- a trial engine,
- a recovery/command/pressure management game,
- a learning loop driven by AAR feedback.

### 2. Current prototype advanced to v0.3.5

The active build now includes:

- split-screen planning/control layout,
- left-side Force Roster / Action Screen,
- center tactical board,
- right-side Mission Coach / Planning Screen,
- phase guidance,
- recommended-action guidance,
- C2 Net Tether logic,
- pressure screening,
- support markers for MG and mortar/indirect support,
- clearer adversary/pressure markers,
- PSG/R7 recovery distinction,
- abstract recovery/repair route,
- AAR categories and next-trial learning direction.

### 3. Key logic corrections made

#### PSG / R7 recovery correction

Earlier builds made the Platoon Sergeant appear to physically execute recovery movement. That was corrected.

Locked logic:

- Platoon Sergeant stays with the operating platoon.
- Platoon Sergeant coordinates recovery.
- R7 and the parts track perform the recovery/repair route.
- R7 handles the active repair cycle.

#### Fire Support / MG correction

Fire Support and MG Base remain abstract pressure-management actions only.

Locked logic:

- Fire Support does not move vehicles.
- MG Base does not move vehicles.
- Both actions only affect pressure/friction markers.
- No real-world targeting or weapon employment is modeled.

#### Guidance correction

The prototype needed more built-in player guidance because the user was still clicking experimentally.

v0.3.5 added:

- Mission Coach,
- phase checklist,
- Coach Select button,
- recommended-action highlighting,
- clearer “select first, then act” flow.

## Current design lock

### Product identity

YAT YAS UP is a **planning and control TTRPG board**, not an arcade shooter.

### Force package

| Asset | Count | Role |
|---|---:|---|
| P7 | 8 | AAV platoon tracks |
| C7 | 1 | Beach C2 support node |
| R7 | 1 | Recovery / repair support track |

### P7 role lock

| Track | Role |
|---|---|
| P7-1 | Platoon Commander / command authority |
| P7-2 | Platoon Sergeant / recovery coordination |
| P7-3 | Troop Carrier 1 |
| P7-4 | Troop Carrier 2 |
| P7-5 | Troop Carrier 3 |
| P7-6 | Troop Carrier 4 |
| P7-7 | C7 chase / support element |
| P7-8 | R7 chase / parts track |

### Locked control points

- Center Beach
- Beach C2 Anchor
- Arrival
- Dismount
- Rally / Organize
- Remount
- Forward Advance
- Recovery Control

### Locked player interaction model

The player should:

1. select a track, marker, or group,
2. issue a phase-valid order,
3. watch the board resolve through abstract friction,
4. use the Mission Coach when unsure,
5. read the AAR,
6. adjust the next run.

## Guardrails reaffirmed

Do not model:

- real tactics,
- weapon employment,
- targeting methods,
- vulnerability exploitation,
- breach methods,
- communications procedures,
- real maintenance procedures,
- casualty simulation,
- squad battle drills,
- field-manual procedures,
- realistic fire support.

All mechanics must stay at the level of:

- pressure,
- friction,
- delay,
- readiness,
- C2 tether,
- screening,
- recovery route,
- abstract repair cycle,
- AAR scoring.

## Current watch items

The project is on the right path, but the next pass should continue improving first-time usability.

Watch items:

- Player still may need clearer next-step guidance.
- Action buttons should explain why they are available or unavailable.
- Roster and selected-unit state must stay easy to read.
- Mission Coach should become more directive without becoming intrusive.
- AAR should provide stronger “change this next time” recommendations.
- The prototype must not drift into real-world tactical detail.
